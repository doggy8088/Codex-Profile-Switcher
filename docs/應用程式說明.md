# Codex Profile Switcher 應用程式說明

## 目的與範圍

Codex Profile Switcher 是一個 macOS 選單列應用程式，用來管理多組 Codex 設定檔與選擇性的檔案式登入快照。它的核心工作是把選定的 profile TOML 寫入 `~/.codex/config.toml`，必要時還原對應的 `auth.json` 快照，最後重新啟動 Codex。

此專案是非官方工具，不直接依賴 OpenAI 或 Codex 的內部 API。它透過本機檔案系統與 macOS AppKit/SwiftUI API 完成操作。

## 專案結構

```text
CodexProfileSwitcher/
  ContentView.swift                 # App 入口、SwiftUI UI、ProfileStore、LoginItemStore
  Assets.xcassets/                  # App icon
CodexProfileSwitcherTests/
  ProfileStoreTests.swift           # ProfileStore 持久化行為測試
Assets/Screenshots/                 # README 圖片
Casks/codex-profile-switcher.rb     # Homebrew cask
.github/workflows/release.yml       # tag release、zip、checksum、cask 更新
Makefile                            # 常用開發指令
README.md                           # 使用者導向說明
AGENTS.md                           # 貢獻者與代理工具指南
```

目前主要程式碼仍集中在 `ContentView.swift`。若未來持續增加功能，優先考慮把 `ProfileStore` 的檔案系統邏輯抽到獨立檔案，讓 UI 與持久化責任分離。

## 架構概覽

應用程式由三個主要層次組成：

| 層次 | 主要型別 | 責任 |
|---|---|---|
| App 入口 | `CodexProfileSwitcherApp`、`AppDelegate` | 建立選單列、管理視窗、避免多重 app 實例 |
| UI | `ProfileMenuView`、`ProfileManagerView`、`ProfileEditorView` | 顯示 profile 清單、編輯內容、執行使用者操作 |
| 狀態與持久化 | `ProfileStore`、`LoginItemStore` | 讀寫 profile、auth snapshot、active state、啟用登入後啟動 |

`ProfileStore` 使用 `@MainActor` 與 `ObservableObject`，讓 UI 狀態更新保持在主執行緒。所有 profile 列表、active profile、狀態訊息都從這個 store 對外發布。

## 核心概念

### Profile

Profile 是儲存在 `~/.codex/profile-switcher/profiles` 的 TOML 檔。檔名會轉成顯示名稱，例如 `Azure-OpenAI.toml` 會顯示為 `Azure OpenAI`。

Profile 的內容會在套用時寫入：

```text
~/.codex/config.toml
```

### Auth Snapshot

Auth snapshot 是 profile 可選擇附帶的登入快照，來源是：

```text
~/.codex/auth.json
```

保存位置是：

```text
~/.codex/profile-switcher/auth/<profile-base-name>.auth.json
```

Auth snapshot 可能包含 Codex access token，必須視同密碼處理。

### Active Profile

Active profile 代表目前 `~/.codex/config.toml` 內容與哪個 profile 相同。程式會透過以下方式判定：

1. 讀取目前 `~/.codex/config.toml`。
2. 與每個 profile 的 TOML 內容比對。
3. 若內容相同，更新 `activeProfileID`。
4. 若無法比對，退回上次保存於 state 檔的 profile ID。

State 檔位於：

```text
~/.codex/profile-switcher/state.json
```

## 主要工作流程

### 建立空白 Profile

1. 使用者點選新增空白 profile。
2. `ProfileStore.createBlankProfile()` 建立預設 TOML 內容。
3. `createProfile(name:contents:)` 產生唯一顯示名稱與安全檔名。
4. TOML 以 `0600` 權限寫入 profiles 目錄。
5. `reload()` 重新讀取 profile 清單。

### 從目前 Codex 設定建立 Profile

1. 使用者點選從目前設定建立 profile。
2. `createProfileFromCurrentConfig()` 讀取 `~/.codex/config.toml`。
3. 寫入 `Current Config.toml` 或自動加尾碼的唯一名稱。
4. 重新載入清單。

### 編輯與重新命名 Profile

1. 使用者在 `ProfileEditorView` 編輯名稱或 TOML。
2. `ProfileStore.update()` 清理顯示名稱。
3. `safeFileName(for:)` 產生安全檔名。
4. `uniqueName()` 同時檢查顯示名稱與檔名，避免 `A/B`、`A:B` 這類正規化後碰撞。
5. 寫入新 TOML 檔。
6. 若檔名改變，移動對應 auth snapshot。
7. 刪除舊 TOML 檔。
8. `reload()` 後回傳重新載入後的 profile，確保 ID 與 UI state 一致。

### 套用 Profile

1. `apply(_:)` 呼叫 `ensureBackup()` 備份目前 `config.toml`。
2. 寫入 profile TOML 到 `~/.codex/config.toml`。
3. 若 profile 有 auth snapshot，呼叫 `applyAuthSnapshotIfAvailable(for:)`。
4. 寫入 active profile state。
5. 重新啟動 Codex。

備份檔名格式：

```text
config.toml.profile-switcher-backup-yyyyMMdd-HHmmss
auth.json.profile-switcher-backup-yyyyMMdd-HHmmss
```

若同一秒內已存在相同檔名，會自動加上 `-2`、`-3` 等尾碼。

### 捕捉目前登入

1. 使用者先讓 Codex 使用檔案式 credential storage。
2. Codex 產生 `~/.codex/auth.json`。
3. 使用者點選 `Capture Current Login`。
4. `captureCurrentAuth(for:)` 讀取目前 auth 檔。
5. 寫入對應 profile 的 auth snapshot，權限為 `0600`。

### 刪除 Profile

刪除前 UI 會顯示確認對話框。若 profile 有 auth snapshot，提示會明確指出登入快照也會一併刪除。

刪除流程：

1. 刪除 profile TOML。
2. 若存在對應 auth snapshot，一併刪除。
3. 若刪除的是 active profile，清空 active state。
4. 重新載入清單。

## 檔案權限與安全模型

下列檔案由程式以私人權限寫入：

| 檔案類型 | 權限 | 原因 |
|---|---:|---|
| Profile TOML | `0600` | 可能含 provider endpoint 或使用者自行填入的敏感設定 |
| `~/.codex/config.toml` | `0600` | 套用後的實際 Codex 設定 |
| `auth.json` snapshot | `0600` | 可能含 access token |
| `config.toml` / `auth.json` 備份 | `0600` | 備份內容可能同樣敏感 |

維護時不要把 profile 內容、auth snapshot 內容或 token 類資料寫入 log、測試 fixture 或文件。

## Release 與 Homebrew 流程

Release workflow 位於 `.github/workflows/release.yml`，觸發條件是推送 `v*` tag。

流程摘要：

1. checkout repository。
2. 若 secrets 存在，匯入 Developer ID certificate。
3. 使用 Xcode Release configuration 建置 app。
4. 若沒有簽章資訊，使用 ad-hoc signing。
5. 使用 `ditto` 打包 app zip。
6. 計算 zip 的 SHA-256。
7. 若 Apple notarization secrets 存在，送 notarization 並 staple。
8. 使用最終 zip checksum 更新 Homebrew cask。
9. 建立 GitHub Release 並上傳 zip 與 `.sha256`。

自動更新 cask 時使用：

```text
git config user.name "github-actions[bot]"
git config user.email "88981+doggy8088@users.noreply.github.com"
```

提交訊息使用暫存檔與 `git commit -F`，符合專案提交規則。

## 開發指令

常用指令：

```sh
make help
make build-unsigned
make test-unsigned
make clean
```

用途：

| 指令 | 用途 |
|---|---|
| `make list` | 列出 Xcode project target、configuration 與 scheme |
| `make build` | 使用 `CONFIGURATION` 建置，預設 Debug |
| `make build-unsigned` | 關閉 code signing 建置，適合一般本機驗證 |
| `make test-unsigned` | 關閉 code signing 執行 XCTest |
| `make clean` | 清理 `.build/DerivedData` |
| `make open` | 開啟 Xcode project |

若本機缺少 `Mac Development` certificate，使用 `make build-unsigned` 與 `make test-unsigned`。

## 測試策略

目前測試位於 `CodexProfileSwitcherTests/ProfileStoreTests.swift`。

已覆蓋行為：

- 新增 profile 後檔案權限為 `0600`。
- 重新命名 profile 後，回傳的 ID 必須存在於 reload 後的 profile 清單。
- 套用重新命名後的 profile，active state 必須可被新 store 還原。
- 同秒連續套用 profile 必須建立唯一備份檔，不可因檔案已存在失敗。

新增功能時，優先補 `ProfileStore` 的 characterization tests。若功能涉及 UI 互動，至少保留可測的 store/service 層邏輯，UI 再做手動驗證。

## 維護注意事項

### ProfileStore 仍偏大

`ProfileStore` 同時負責路徑、檔案 IO、備份、auth snapshot、active state 與重啟 Codex。若要重構，建議順序如下：

1. 先補足目前行為測試。
2. 抽出檔案系統 adapter 或 storage service。
3. 讓 `ProfileStore` 只負責狀態協調與 UI-facing API。
4. 保持公開方法名稱與 UI 呼叫點穩定，降低回歸風險。

### 檔名與顯示名稱不可混用

顯示名稱可能含 `/`、`:` 等字元，寫入檔案前必須走 `safeFileName(for:)`。任何新增的 profile 操作都應同時考慮：

- 顯示名稱唯一性。
- 安全檔名唯一性。
- auth snapshot 是否需同步移動或刪除。
- active profile ID 是否仍有效。

### 不要假設 Auth Snapshot 一定存在

Profile 可以只有 TOML，沒有登入快照。所有 auth snapshot 操作都必須先檢查檔案是否存在。

### 重啟 Codex 是副作用

`ProfileStore` 建構子支援 `restartsCodexOnApply: false`，主要給測試使用。測試不應真的啟動或關閉 `/Applications/Codex.app`。

### 刪除是破壞性操作

刪除 profile 會刪除 TOML 與可能存在的 auth snapshot。UI 必須保留確認流程，避免誤刪登入快照。

## 常見問題排查

### `xcodebuild` 找不到簽章憑證

使用：

```sh
make build-unsigned
make test-unsigned
```

### CoreSimulator 版本警告

此專案是 macOS app。若 `xcodebuild` 顯示 CoreSimulator out-of-date，但 macOS target 測試或建置成功，通常不影響本專案驗證。

### Codex 沒有產生 `auth.json`

確認 Codex 設定包含：

```toml
cli_auth_credentials_store = "file"
```

然後重新登入 Codex，再回到 app 捕捉目前登入。
